# CUBRID 원격 백업 방식 A / B 비교

`cubrid backupdb` 결과를 다른 IP 백업 서버로 전송하는 두 가지 방식(A: rbk 재개형, B: ssh 스트리밍)의 차이와 선택 기준 정리.

## 1. 방식 A, B 비교표

| 항목 | 방식 A (rbk 재개형) | 방식 B (ssh 스트리밍) |
|------|------|------|
| 전송 경로 | FIFO → 로컬 스풀 → TCP(9099) → rbk_listener | FIFO → ssh(22) → cat > 파일 |
| 로컬 디스크 | 백업 크기만큼 스풀 필요(약 35GB) | 거의 불필요 |
| 수신측 준비 | rbk_listener 빌드/실행 + 전송 포트 개방 | ssh만 있으면 됨(전용 프로그램 불필요) |
| 네트워크 순단 | 재접속 후 이어받기(중단 지점부터) | 재개 불가 → 처음부터 다시 |
| 원격 장애(무응답) | timeout 초과 시 backupdb 정지 + 오류 로그(조건2) | 자체 감지 없음(ssh가 멈추거나 늘어짐) |
| 운영 DB 커밋 지연 | 스풀이 흡수 → 거의 없음(지연 회피) | 망이 느리면 지연 발생 가능 |
| 구성 복잡도 | 높음(프로그램·포트·스풀) | 낮음(ssh 한 줄) |

## 2. 방식 A: 대용량 + 불안정/원거리 망, 운영 지연 절대 회피

핵심은 "방식 A는 로컬 스풀이라는 완충 장치 덕분에 (1) 운영 커밋이 안 밀리고 (2) 끊겨도 이어받고 (3) 오래 죽으면 안전하게 백업을 멈출 수 있다"는 점이고, 그 대가로 로컬 공간과 수신측 프로그램, 포트가 필요하다는 것입니다.

전송 받는 장비/네트워크가 backupdb 속도를 잘 따라가면, 포워더가 스풀에 쓰자마자 곧바로 원격으로 빠져나가므로 스풀에 남는 양은 아주 작게(거의 실시간 통과) 유지됩니다. 반대로 원격이 느리거나 잠깐 끊기면, 못 보낸 데이터가 스풀에 쌓여 최대 백업 크기(약 35GB)까지 커질 수 있습니다. 즉 로컬 디스크 스풀(파이프 파일) 크기는 고정이 아니라 "backupdb 생산 속도 − 원격 전송 속도"의 격차입니다.

## 3. 방식 B: 안정·빠른 망, 로컬 여유 공간 부족, 간단히

핵심은 "방식 B는 완충 장치(스풀)가 없어 backupdb가 만든 데이터를 곧바로 ssh로 원격에 흘려보낸다"는 점입니다. 그 덕분에 (1) 로컬에 백업 사본을 안 만들어 디스크 공간이 거의 안 들고 (2) 수신측은 ssh만 있으면 되어 전용 프로그램·전송 포트가 필요 없으며 (3) 명령 한 줄로 끝날 만큼 단순합니다. 대신 backupdb가 원격 장비에 직접 묶인다는 것이 대가입니다.

동작을 보면, FIFO(nhidb_fifo)는 데이터를 저장하지 않는 임시 파이프라서, backupdb가 쓰는 속도와 원격이 받아 쓰는 속도가 강제로 같아집니다. 즉 원격/네트워크가 느려지면 파이프가 막히고 backupdb도 그만큼 기다립니다. 여기서 방식 A와 반대되는 세 가지 한계가 나옵니다. (1) 망이 느리면 파이프가 막혀 backupdb가 대기 → 운영 DB 커밋이 지연될 수 있고, (2) 전송 중 끊기면 이어받기가 없어 처음부터 다시 해야 하며, (3) 원격이 오래 무응답이어도 이를 감지해 안전하게 멈추는 watchdog이 없어 ssh가 늘어지거나 멈춘 채로 있습니다.

정리하면, 방식 B는 "완충이 없어 단순하고 가볍지만, 그만큼 망 상태에 직접 노출된다"는 것이고, 그래서 안정적이고 빠른 망 + 로컬 공간이 부족한 상황에 적합합니다. 반대로 대용량·불안정·원거리 망이거나 운영 지연을 절대 피해야 하면 방식 A가 맞습니다.

## 참고: 실측 (2026-07-23, 운영 nhidb, cubrid 계정)

운영 중 nhidb(약 168GB, Full/Level 0 온라인 백업)를 `192.168.7.39:/home/cubrid/db_backupdb` 로 전송.

| 방식 | 결과 | 소요 | 크기 |
|------|------|------|------|
| A (rbk 재개형) | rc=0 | 약 4분 33초 | 36,721,144,832B (약 35GB) |
| B (ssh 스트리밍) | backupdb rc=0, ssh rc=0 | 약 4분 17초 | 36,721,144,832B (약 35GB) |

- 두 방식 모두 파일명은 backupdb 기본 규칙 `nhidb_bk0v000`, 회차 디렉터리 `nhidb_<TS>/` 안에 저장.
- 복원: `cubrid restoredb -B /home/cubrid/db_backupdb/nhidb_<TS> nhidb`.
- 백업 중 nhidb 운영은 계속 유지됨.
