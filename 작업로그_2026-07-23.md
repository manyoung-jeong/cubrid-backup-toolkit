# 작업 로그 - 2026-07-23 (CUBRID 원격 백업 툴킷)

운영 중 nhidb(약 168GB)를 다른 IP 백업 서버로 전송하는 파이프 백업 툴킷을 정비/검증했다.

## 1. 백업 대상 계정/경로/파일명 정비
- 백업 서버 계정을 root → **cubrid** 로 전환, 경로 `/home/cubrid/db_backupdb`.
- FIFO/스풀에 실행 시각 suffix 부여(유니크) → 반복/동시 실행 충돌 방지.
- 저장 파일명을 backupdb 기본 규칙 `<DB>_bk0v000` 으로 변경, 회차 디렉터리 `<DB>_<TS>/` 안에 저장(복원 편의: `restoredb -B <디렉터리>`).
- `run_nhidb_A.sh` 추가: 방식 A 원스텝(리스너 기동→포트확인→백업→결과확인).

## 2. 방식 A / B 비교 문서화
- `백업방식_A_B_비교.md` 작성: 비교표 + 선택 기준 + 실측.
- 핵심: A는 로컬 스풀 완충으로 커밋지연 회피+이어받기+무응답 정지(대가: 로컬 공간/수신 프로그램/포트). B는 완충 없이 단순하나 망 상태에 민감.

## 3. 실서비스 배포 정리
- 필요한 파일만 담은 배포 tar `cubrid_backup_toolkit.tar.gz` 생성.
- README 를 "tar 풀기 → 빌드 → 수신/송신 설정 → 실행 → 검증" 배포 절차로 재작성.
- 불필요 파일 제거: 문서 생성기(gen_*.py), mock 자체시험(e2e_devices_test.sh). (git 이력에 보존)

## 4. backupdb 옵션 파이프 백업 검증
- 스크래치 DB로 각 옵션을 `-D <FIFO>` 로 실제 실행해 확인.
- 사용 가능(rc=0): `-l 0/1`(증분 포함), `-C`, `-S`(DB 정지 시), `--no-check`, `-t`, `--no-compress`(스트림 3MB→173MB 증가), `--sleep-msecs`, `-r`, `-o`.
- **사용 불가: `-k, --separate-keys`** — CUBRID 가 "Cannot use FIFO file as the destination-path with separate-key" 로 거부.
- 사용 가능한 옵션을 모두 넣은 예제를 README/문서/docx(5.4)에 반영.

## 5. 백업본 검증(복원) 방법
- README 검증 안내를 checkdb → **`csql -S <DBname>`**(standalone 접속으로 오픈 확인)로 변경.

## 6. 로컬 스풀 최대 크기 제한 기능 (신규)
- 요구: 스풀 최대 크기를 사용자가 설정(기본 2G), 초과 시 오류 로그 남기고 백업 종료.
- 구현: rbk_forward.c 를 **링 버퍼**로 개선(이미 전송+안전마진 구간 재사용) → 로컬 디스크가 SPOOL_MAX 를 절대 넘지 않음. 미전송 백로그가 한도 초과 시 오류 로그 + `exit(3)`, 래퍼가 backupdb 정지.
- 설정: `SPOOL_MAX`(예 2G/512M/바이트), 기본 2G. 반환코드 3 추가.
- 순단 이어받기 안전: 보낸 지점에서 32MB 마진을 링에 남겨 재접속 오프셋부터 재전송 보장.

## 7. 검증 결과
- 단독(forwarder/listener): 정상 전송 무결성 cmp 일치(rc=0), 스풀 초과 시 rc=3 + 오류 로그.
- **실측(nhidb, SPOOL_MAX=2G)**: 18:13:16→18:17:14(약 4분), total=36,721,144,832B(약 35GB, 바이트 일치), rc=0. 스풀 파일 최대 2,147,483,648B(=2G) 유지 — 상한 준수하며 35GB 완주.
